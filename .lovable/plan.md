# Fix: AssemblyAI "auto_chapters not available in this language" (TV)

## Root cause

`supabase/functions/process-tv-with-qwen/index.ts` (line 754–769) creates the AssemblyAI transcription with:

```ts
language_code: 'es',
auto_chapters: true,
iab_categories: true,
```

AssemblyAI's `auto_chapters` (and `iab_categories`) are English-only models. With `language_code: 'es'` the API returns 400:

```
"The following models are not available in this language: auto_chapters"
```

This blocks every TV transcription at the "Creating AssemblyAI transcription job" step (visible in the logs for both `req_1781102650080` and `req_1781102728170`). It is a pre-existing config bug, not caused by the 1.4 prompt-only change — but it's the active blocker right now.

We don't actually consume `auto_chapters` or `iab_categories` output anywhere downstream: the TV analysis is generated by Qwen/Gemini from the transcript text + our own prompt, and chapters/topics come from our own segmentation in `analysis_*` fields.

## Change

Single-file edit in `supabase/functions/process-tv-with-qwen/index.ts`:

- Remove `auto_chapters: true`
- Remove `iab_categories: true`
- Keep `language_code: 'es'`, `speaker_labels: true`, `entity_detection: true`, `speech_model`

Then deploy `process-tv-with-qwen` and verify by re-running a TV upload (status should advance past "Creating AssemblyAI transcription job" without a 400).

## Out of scope

- No prompt changes (1.4 work stays as-is)
- No change to `transcribe-video/transcription-service.ts` (radio path); we only touch the TV path that's currently failing. If radio is also affected we'll handle as a follow-up.
- No retry/backoff or model-selection logic changes.
